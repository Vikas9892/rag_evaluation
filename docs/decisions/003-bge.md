# ADR 003 — Embedding Model Selection (BAAI/bge-small-en-v1.5)

**Status:** Accepted  
**Date:** 2026-07

---

## Context

The embedding model determines the quality of semantic similarity between queries and
chunks.  Key constraints: runs on CPU in CI and Lambda (no GPU), fits in Lambda's
250 MB deployment limit, and achieves competitive retrieval quality on MTEB benchmarks.

## Decision

Use `BAAI/bge-small-en-v1.5` via `sentence-transformers`.  Embed with
`normalize_embeddings=True` (BGE instruction-tuned models require this) and store
384-dimensional float32 vectors.

## Rationale

BGE-small strikes the best accuracy/size balance in the sub-100 MB class:

| Model | Params | Dim | MTEB avg | Size |
|-------|--------|-----|----------|------|
| bge-small-en-v1.5 | 33 M | 384 | 62.2 | 130 MB |
| bge-base-en-v1.5  | 109 M| 768 | 64.2 | 430 MB |
| all-MiniLM-L6-v2  | 22 M | 384 | 56.3 | 90 MB  |
| text-embedding-ada-002 | — | 1536 | 61.0 | API |

BGE-small outperforms all-MiniLM-L6-v2 by ~6 MTEB points while being similarly sized.
BGE-base is better but exceeds Lambda's 250 MB unzipped package limit.
OpenAI ada-002 would require an API call per chunk during ingestion — slow, costly, and
not reproducible offline.

**Why `normalize_embeddings=True`?**  BGE was trained with query/passage prefixes and
cosine distance.  L2 normalisation before inner-product search makes IP ≡ cosine
similarity, which is what the model was trained to optimise.

## Consequences

- 130 MB model weights — must be pre-cached or stored in S3 / Lambda Layer for production.
- English-only; multilingual requires `bge-m3`.
- Embedding is the bottleneck during ingestion (~6.7 ms/chunk on CPU).

## Alternatives Considered

| Option | Rejected because |
|--------|-----------------|
| all-MiniLM-L6-v2 | Lower MTEB score; no clear size advantage |
| bge-base-en-v1.5 | Exceeds Lambda package limit (430 MB) |
| text-embedding-ada-002 | API dependency; offline ingestion impossible |
| instructor-xl | 1.3 B params; far too slow on CPU |
